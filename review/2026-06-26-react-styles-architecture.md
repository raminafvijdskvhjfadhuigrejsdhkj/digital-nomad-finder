## A. Функциональные баги (сначала это)

### [3] Краш страницы «Favorites» 🔴 высокий
- **Файл:** `src/pages/Favorites.jsx:20`
- **Что:** `key={country.names.common}` — обращение к `country.names`, которого
  в данных нет (после смены API объект страны имеет форму `name.common`).
  Проверено вживую: добавляешь страну в избранное, заходишь на `/favorites` —
  белый экран и в консоли `TypeError: Cannot read properties of undefined
  (reading 'common')`.
- **Почему:** одна страница приложения полностью падает. Это худший тип бага —
  не косметика, а отказ функции, которую ты же и сделал.
- **Как:** обращайся к данным так же, как в рабочих местах (`name?.common`).
  И загляни в пункт [6] — корень в том, что у тебя нет единой формы данных,
  и приходится в каждом файле гадать `names` или `name`.

### [4] Фильтр по региону не работает
- **Файлы:** `src/App.jsx` (маппинг) + `src/components/RegionFilter.jsx`
- **Что:** при загрузке каждой стране присваивается `region: "All"`, а в фильтре
  опции `Europe` / `Asia`. Ни одна страна под них не подходит — выбор региона
  всегда даёт пустой список.
- **Почему:** «функция, которая молча ничего не делает» хуже отсутствующей —
  пользователь думает, что сломано всё приложение.
- **Как:** реши честно. Либо нужен источник данных, где у стран реальный регион
  (тогда фильтр оживёт), либо убери фильтр до тех пор, пока данных нет.
  Главное — UI не должен обещать то, чего не делает.

### [5] «Visa Type» и кнопка «Search» — пустышки
- **Файлы:** `src/components/VisaFilter.jsx`, `src/pages/Home.jsx` (SearchButton)
- **Что:** выбранная виза хранится только внутри `VisaFilter` и никуда не влияет.
  Кнопка `Search` без обработчика — поиск и так работает на вводе текста.
- **Почему:** то же, что [4] — мнимая функциональность путает.
- **Как:** либо подними состояние визы наверх и применяй в фильтрации (см. [9]),
  либо убери до реализации. Кнопку `Search` — либо повесь смысл, либо убери.

## B. Архитектура и данные

### [6] Нет единого источника правды для модели страны
- **Файлы:** `src/App.jsx`, `src/components/CountryCard.jsx`, `Favorites.jsx`
- **Что:** по коду разбросано `country.names?.common || country.name?.common`,
  `getCountryName` написан дважды (в `App.jsx` и в `CountryCard.jsx`) разными
  способами, где-то `capital`, где-то `capitals[0].name`. Это следы двух разных
  API, оставшиеся в коде.
- **Почему:** каждый новый компонент вынужден гадать форму данных — отсюда баг [3].
  Дублирование `getCountryName` нарушает DRY: поправишь в одном месте, забудешь
  в другом.
- **Как:** нормализуй данные **один раз** — там, где приходит ответ API (в `App`).
  Приведи каждую страну к одной форме (например `{ name, capital, region,
  population, image }`), и дальше по всему коду обращайся к ней напрямую, без
  цепочек `?.` и без `||`. Общие хелперы вынеси в отдельный файл (напр. `src/utils/`).

### [7] Дублирование роутов `/` и `/countries`
- **Файл:** `src/App.jsx`
- **Что:** оба роута рендерят один и тот же `<Home>` с полным копипастом всех
  пропсов.
- **Почему:** любой новый проп придётся добавлять в двух местах — легко
  рассинхронить.
- **Как:** вынеси элемент в переменную и переиспользуй, либо пересмотри — нужен ли
  вообще отдельный `/countries`, если он идентичен главной.

### [8] `RegionFilter` и `VisaFilter` — почти одинаковые компоненты
- **Файлы:** `src/components/RegionFilter.jsx`, `src/components/VisaFilter.jsx`
- **Что:** два дропдауна на ~90% совпадают (стили + логика open/select).
- **Почему:** дублирование разрастается — каждую правку дропдауна надо делать дважды.
- **Как:** сделай один переиспользуемый `<Dropdown options value onChange />`
  и используй его и для региона, и для визы. Это хорошее упражнение на
  «компонент как переиспользуемый кубик».

## C. React-практики

### [9] Состояние визы «заперто» в дочернем компоненте
- **Что:** `VisaFilter` держит `useState` визы у себя, и наружу оно не выходит.
- **Почему:** фильтровать страны по визе невозможно — родитель про выбор не знает.
- **Как:** загугли «lifting state up». Состояние, от которого зависит несколько
  компонентов, должно жить выше — в `App`, рядом с `search` и `region`, и
  передаваться вниз. Сравни с тем, как уже сделаны `search`/`region`.

### [10] Дропдаун не закрывается по клику снаружи + нет клавиатуры
- **Файлы:** `RegionFilter.jsx`, `VisaFilter.jsx`
- **Что:** открыл дропдаун, кликнул мимо — он остаётся открытым. С клавиатуры не
  управляется.
- **Почему:** это базовое ожидаемое поведение любого селекта; без него UX «дёшевый».
- **Как:** посмотри в сторону закрытия по клику вне (ref + слушатель на `document`,
  с очисткой в `useEffect`). Это хороший повод понять `useRef` и побочные эффекты.

### [11] Логика экрана загрузки спорная
- **Файл:** `src/App.jsx`
- **Что:** `Loading` показывается только если запрос длится дольше 2 сек. Первые
  2 секунды пользователь видит пустую страницу с заголовком и без карточек.
- **Почему:** обычно индикатор загрузки показывают сразу, иначе экран выглядит
  «сломанным/пустым».
- **Как:** обдумай, какое поведение ты хотел. Если цель — не мигать спиннером при
  быстрых ответах, опиши это решение; если нет — показывай загрузку сразу, пока
  `loading === true`.

## D. Стили и разметка

### [12] Цвета захардкожены вместо темы
- **Файлы:** почти везде (`Home.jsx`, `CountryCard.jsx`, `Loading.jsx`, ...)
- **Что:** `#adc6ff`, `#002e6a`, `#ff4d6d`, `#8c909f`, `rgba(173,198,255,...)`,
  `#050505` раскиданы по компонентам, хотя есть `theme.js`.
- **Почему:** «магические» цвета сложно менять и легко рассинхронить между тёмной
  и светлой темами; в `Loading` фон вообще всегда чёрный и не реагирует на тему.
- **Как:** перенеси палитру в `theme.js` и обращайся через `${({theme}) => ...}`.
  Цель — поменять цвет в одном месте, а не искать по всему проекту.

### [13] Страница «Favorites» выпадает из общего стиля
- **Файл:** `src/pages/Favorites.jsx`
- **Что:** свёрстана инлайн-стилями (`style={{...}}`), без `styled-components`,
  без `Navbar`, без темы (фон не темнеет), сетка жёстко в 4 колонки без адаптива.
- **Почему:** несогласованность с остальным приложением; попав на `/favorites`,
  пользователь не может вернуться (нет навигации); на мобильном вёрстка поедет.
- **Как:** приведи страницу к тому же подходу, что и `Home` — `styled-components`,
  общий `Navbar`, тема, адаптивная сетка (можно переиспользовать готовую сетку
  карточек).

### [14] Доступность: `label` не связан с полем
- **Файл:** `src/pages/Home.jsx`
- **Что:** `<Label htmlFor="search">` / `htmlFor="region"` есть, но у самих полей
  (`SearchBar`, `RegionFilter`) нет соответствующего `id`. У «Visa Type» `htmlFor`
  вовсе нет.
- **Почему:** клик по подписи не фокусирует поле, скринридеры не связывают их —
  это база доступности форм.
- **Как:** прокинь `id` в поле, совпадающий с `htmlFor`. Заодно почитай, почему
  доступность форм важна.

## E. Требования к разработке и безопасность

### [15] Нет состояний ошибки/пустого результата в интерфейсе
- **Файл:** `src/App.jsx` (`fetch().catch`)
- **Что:** при ошибке запроса — только `console.error`, пользователь видит пустую
  сетку. Нет сообщения «не удалось загрузить» и нет «ничего не найдено» при
  пустом результате фильтра.
- **Почему:** надёжный фронтенд всегда обрабатывает три состояния: загрузка /
  ошибка / пусто. Иначе сбой выглядит как «приложение зависло».
- **Как:** заведи состояние ошибки и показывай понятное сообщение; добавь текст
  для пустого списка.

### [16] Чтение `localStorage` без защиты
- **Файлы:** `src/App.jsx`, `src/main.jsx`
- **Что:** `JSON.parse(localStorage.getItem(...))` без `try/catch`.
- **Почему:** если данные в хранилище повредятся (или туда попадёт мусор),
  `JSON.parse` бросит исключение и приложение не запустится.
- **Как:** оберни чтение в `try/catch` и возвращай значение по умолчанию при сбое.

### [17] README — дефолтный шаблон Vite
- **Файл:** `README.md`
- **Что:** README остался стандартным текстом из шаблона, проект не описан.
- **Почему:** README — лицо проекта; по нему понимают, что это, как запустить,
  какой стек. Это часть «требований к разработке».
- **Как:** напиши коротко: что за приложение, стек, как запустить (`npm i`,
  `npm run dev`), какие переменные окружения нужны (см. пункт про `.env` из
  итерации 1).
